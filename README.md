# Реалізація GOF Патернів Проєктування на TypeScript

Цей проєкт демонструє використання 6 класичних патернів проєктування (Gang of Four) за допомогою мови TypeScript. Проєкт поділений на три основні групи: породжувальні, структурні та поведінкові патерни.

## Як запустити проєкт

1. Переконайтеся, що у вас встановлений [Node.js](https://nodejs.org/).
2. Встановіть залежності:
   ```bash
   npm install

## Опис реалізованих патернів та архітектурних покращень

### Породжувальні патерни (Creational)
1. **Singleton (Одинак)**
  * **Проблема:** Потрібно обмежити створення об'єкта одним екземпляром (наприклад, БД).
  * **Рішення:** Робить конструктор приватним і надає статичний метод для доступу.
  * **Покращення:** Усуває неконтрольоване створення "важких" об'єктів, на відміну від анти-патерну глобальних змінних.
2. **Factory Method (Фабричний метод)**
  * **Проблема:** Делегування створення об'єктів підкласам без прив'язки до конкретних класів.
  * **Рішення:** Базовий клас викликає абстрактний метод `createTransport()`.
  * **Покращення:** Усуває анти-патерн "Божественний метод" з нескінченними `switch-case` конструкціями при додаванні нових типів.

### Структурні патерни (Structural)
3. **Adapter (Адаптер)**
  * **Проблема:** Несумісність інтерфейсів власного коду та сторонньої бібліотеки.
  * **Рішення:** Створення класу-обгортки, який перетворює виклики.
  * **Покращення:** Дозволяє уникнути переписування робочого бізнес-коду під новий інструмент (зберігає принцип відкритості/закритості - OCP).
4. **Decorator (Декоратор)**
  * **Проблема:** Динамічне додавання властивостей без розростання ієрархії спадкування.
  * **Рішення:** Обгортання базового об'єкта в декоратори (напр., Email -> SMS -> Slack).
  * **Покращення:** Усуває анти-патерн "Вибух комбінацій підкласів" (Subclass Explosion).

### Поведінкові патерни (Behavioral)
5. **Strategy (Стратегія)**
  * **Проблема:** Необхідність перемикати алгоритми (поведінку) на льоту.
  * **Рішення:** Інкапсуляція алгоритмів у власні класи з єдиним інтерфейсом.
  * **Покращення:** Знищує умовну логіку (`if/else`) всередині головного класу, розділяючи відповідальність (SRP).
6. **Observer (Спостерігач)**
  * **Проблема:** Необхідність повідомляти множину об'єктів про зміни стану іншого об'єкта.
  * **Рішення:** Механізм підписки та розсилки сповіщень.
  * **Покращення:** Позбавляє жорсткої зв'язності (Tight Coupling), коли головний клас знає про деталі реалізації залежних класів.

## Висновки
Використання GOF патернів у TypeScript разом зі строгою типізацією значно покращує масштабованість та підтримуваність архітектури. Це допомагає дотримуватися принципів SOLID, роблячи код гнучким до змін бізнес-вимог.
